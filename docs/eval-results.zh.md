# Mainline 评测报告

## TL;DR

我们设计了 8 个工程场景，让 AI coding agent 分别在两种模式下完成任务：

- **code-first（只看代码）**：拿到任务描述 + 相关代码文件。
- **intent-first（带历史意图）**：额外读取 Mainline 记录的历史意图、决策、风险和 anti-patterns。

在 3 轮 Claude Sonnet 4 运行中，code-first 在 2/8 个场景里稳定犯错，共触发 9 次 forbidden-list violation；intent-first 为 0。

|  | code-first（只看代码） | intent-first（带历史意图） |
|---|---:|---:|
| 任务级失败场景 | 2/8 | 0/8 |
| forbidden violation 总数 | ❌ 9 | ✅ 0 |
| 三轮一致性 | 同样两个场景反复失败 | 三轮均未违反禁令 |

差异集中在两类场景：

1. **已放弃方案仍残留在代码中**：agent 看到半成品实现，倾向于继续补全。
2. **新旧实现并存，但旧方案已被替代**：agent 看到两套代码，倾向于两边都改。

这两类错误的共同点是：**正确答案不在代码里，而在团队的决策历史里。**

> ⚠️ 这不是最终公开 benchmark，而是第一轮受控实验。结果显示方向性信号，
> 后续需要多模型、多 seed、LLM-as-judge 自动评分验证。

---

## 为什么做这个评测

Mainline 的核心主张是：

> AI coding agent 如果能在改代码前看到团队的历史决策，就能避免一些"代码看起来该改、
> 但实际上不该改"的错误。

这个主张很直觉，但需要证据。光说"历史上下文有帮助"不够——得实际跑一遍，
看在哪些场景真的有帮助，哪些场景没区别。

---

## 评测怎么做的

### 概念

把同一个编程任务交给两种 agent：

- **只看代码的 agent（code-first）**：拿到任务描述 + 相关代码文件。跟一个普通 coding agent 一样。
- **带历史意图的 agent（intent-first）**：除了上面的东西，还额外拿到 Mainline 存储的
  团队历史意图——哪些方案被试过然后放弃了、哪些决策替代了旧决策、哪些约定只存在于文档里。

然后看：谁违反了团队约束？谁做了不该做的事？

### 用例设计

8 个场景，覆盖真实工程中常见的"隐性约束"：

| 场景 | 模拟的情况 |
|---|---|
| `abandoned-approach` | Redis 缓存方案写到一半后因复制延迟问题被放弃 |
| `superseded-decision` | CSV 导出被 Parquet 取代，但旧代码还在且有流量 |
| `auth-migration` | 认证模块正在从 session 迁移到 JWT |
| `stale-intent` | 一条很久以前的性能调优决策，环境已变 |
| `billing-boundary` | 计费模块有严格的访问边界规则 |
| `risk-aware-tests` | 团队对某个模块有强制回归测试要求 |
| `docs-only-intent` | 术语规范只存在于文档 commit 中 |
| `refactor-cross-file` | 跨文件重构中必须保持接口签名 |

每个场景都有一份**禁令清单**（agent 不应该做的事）。违反禁令 = 犯错。

### 执行方式

- **模型：** Claude Sonnet 4
- **方法：** 生成 6 个独立 agent（3 个只看代码、3 个带历史意图），各自独立处理全部 8 个场景
- **评分：** 由人工语义判读——agent 是在"提议做被禁止的事"还是"提到它但明确拒绝了"

为什么不用自动评分？因为子串匹配会把"我**不会**这么做"也判定为违规（详见下文）。

## 详细结果

### 第一层：检索是否工作

在对照之前，先确认一个前置条件：Mainline 的检索能不能把历史约束取出来送到 agent 面前？

| 场景 | 结果 |
|---|---|
| `auth-migration` | ✅ 命中 |
| `abandoned-approach` | ✅ 命中 |
| `superseded-decision` | ✅ 命中 |
| `stale-intent` | ✅ 命中 |
| `billing-boundary` | ✅ 命中 |
| `risk-aware-tests` | ✅ 命中 |
| `docs-only-intent` | ✅ 命中 |
| `refactor-cross-file` | ✅ 命中 |

**8/8 通过。** 检索没有漏掉任何关键约束。

### 第二层：agent 真的接住了吗？（活跑 3 轮）

| 场景 | 只看代码（3 轮） | 带历史意图（3 轮） | 结论 |
|---|---|---|---|
| `abandoned-approach` | ❌❌❌ 三轮都犯 | ✅✅✅ 零失误 | **历史意图救命** |
| `superseded-decision` | ❌❌❌❌❌❌ 三轮共 6 次 | ✅✅✅ 零失误 | **历史意图救命** |
| `auth-migration` | ✅ | ✅ | 平手——代码信号够了 |
| `stale-intent` | ✅ | ✅ | 平手 |
| `billing-boundary` | ✅ | ✅ | 平手 |
| `risk-aware-tests` | ✅ | ✅ | 平手 |
| `docs-only-intent` | ✅ | ✅ | 平手 |
| `refactor-cross-file` | ✅ | ✅ | 平手 |

汇总：

```
只看代码：   ❌ 9 次犯错（forbidden violation 计数，非任务计数）
带历史意图： ✅ 0 次犯错
```

> **关于计数方式：** 这里的"犯错次数"按 forbidden-list violation 计数，不按任务计数。
> 例如 `superseded-decision` 每轮违反了两条禁令（添加 CSV 导出字段 + 使用旧 CSV 格式），
> 因此 3 轮共产生 6 次 violation。`abandoned-approach` 每轮 1 次，3 轮共 3 次。合计 9。

### 犯错细节

**abandoned-approach — 每轮都错**

任务是"优化 session 存储性能"。代码里有一个 redis.go 写到 60%，docker-compose
里有 redis service，到处都是 TODO。

- 只看代码的 agent：三轮都提议"继续完成 Redis 迁移"。看起来完全合理——代码摆在那，写都写了一半。
- 带历史意图的 agent：看到这条记录："此方案因复制延迟在高负载下不可接受而放弃"，三轮都拒绝走 Redis 路线。

**superseded-decision — 每轮都错，而且错两个地方**

任务是"给导出端点加一个新字段"。项目里 CSV 和 Parquet 两套导出都在跑，CSV 有一行
`// deprecated` 注释但仍有流量。

- 只看代码的 agent：三轮都把新字段加到 CSV 和 Parquet 两边。合理推断——deprecated 不等于 dead。
- 带历史意图的 agent：看到"CSV 导出已被 Parquet 替代，不再接受新字段"，三轮都只改 Parquet。

---

## 为什么这两个错误在三轮中稳定复现？

不是巧合。是因为这两个场景的代码**结构性地引导 agent 犯错**：

- 一个写到一半的实现 + docker-compose 配置 = "帮我完成它"的强信号
- 一个有流量的端点 + deprecated 注释 ≠ "别碰它"的强信号

这不是 prompt 不好，也不是模型不聪明。问题在于代码只能展示当前残留状态，
无法表达团队为什么保留、放弃或替代某条路径。后者只有决策历史能提供。

在本次 3 轮实验中，这两个场景稳定复现了同样的失败模式。
这个优势在本次实验中稳定复现，值得进一步扩大样本验证。

---

## 带历史意图的 agent 不只是"不犯错"

回放实验还观察到一个更有趣的行为：带历史意图的 agent 不仅避开了禁区，
还**主动引用约束，解释自己为什么不这么做**。

8 个 fixture 共定义了 18 个 forbidden constraints（每个 fixture 有 1-4 条禁令项，
覆盖该场景下 agent 不应该做的所有行为）。在回放实验的 3 轮运行中，
intent-first agent 对这 18 个禁令项都表现为明确引用并拒绝（declined-with-reference），
而不是无声地绕开。这比单纯"没犯错"更强——它意味着 agent 建立了审计链路：

1. ✅ 检索到了约束
2. ✅ 理解了约束内容
3. ✅ 做出了正确选择
4. ✅ 解释了决策理由

---

## 自动评分器为什么不好使

最初用子串匹配做自动评分——看 agent 输出里有没有出现被禁止的关键词。

结果完全不可用：当 intent-first agent 说"我**不会**使用 Redis 方案，因为它之前
因复制延迟被放弃了"时，`Redis` 这个子串出现了，评分器就判定为违规。

这让 intent-first 在自动评分下反而比 code-first"更差"——方向完全反了。

解决方案是语义评分：不是看有没有出现关键词，而是判断 agent 是在**提议做这件事**
还是在**拒绝做这件事**。目前由人工判读，后续会用 LLM-as-judge 自动化。

---

## 这个评测说明了什么

### 说明了的

✅ 当正确答案不在代码里（而在团队决策历史中）时，带历史意图的 agent 能避免犯错。

✅ 这个优势在本次 3 轮实验中稳定复现，不是偶然信号。

✅ Mainline 的检索确实能把相关约束送到 agent 面前（8/8 命中）。

### 没说明的

⚠️ 这不是"intent-first 在所有任务上都更好"。8 个场景中只有 2 个有区别，
其余 6 个代码信号本身就足够了。

⚠️ `docs-only-intent` 在本次 Sonnet 4 code-first 运行中没有触发 violation，
但它仍是 Mainline 设计要覆盖的场景：规则可能不在代码里，而在文档或历史 intent 中。
回放实验（Opus 4.6）中此场景有失败，说明模型差异会影响结果。

⚠️ 只测了一个模型（Claude Sonnet 4），其他模型可能表现不同。

⚠️ N=3 轮，统计学上仍是小样本（虽然一致性很强）。

⚠️ 场景是人工构造的，不是从真实代码库抽样。

### Mainline 真正解决的问题

不是"让 agent 更聪明"，而是**补上代码无法表达的那部分知识**：

| 知识类型 | 代码能表达？ | Intent 能表达？ |
|---|---|---|
| 当前架构 | ✅ | — |
| 已放弃的方案和原因 | ❌ | ✅ |
| 已被替代的旧决策 | 勉强（deprecated 注释） | ✅（明确的替代关系） |
| 跨团队的口头约定 | ❌ | ✅ |
| 测试覆盖的约束 | ✅ | — |

## 一句话总结

> 代码告诉 agent「现在是什么样」。历史意图告诉 agent「哪些旧路已经失败、
> 哪些决策已经过期」。Mainline 补的就是这个缺口——而评测证明这个缺口确实会让 agent 犯错。
